# Changelog — Branch `copilot/fix-vercel-build-error`

All commits listed below were made to fix `next build` failing on Vercel with `Command "pnpm run build" exited with 1`. Each section covers one commit in chronological order.

---

## Commit 1 — `38b4fa4` — 2026-03-26T12:24:34

**Message:** Fix TypeScript build errors causing Vercel deployment failure

### Problem

Running `next build` on Vercel triggered multiple TypeScript errors during the type-checking phase. The build could not complete.

### Changes

#### 1. `components/language-provider.tsx` — lines 76–78, 119–126

**What changed:** Added 11 missing keys to the `Translations` interface.

**Why:** Other components were already using these translation keys (e.g. `t.login_page.error_no_account`), but the TypeScript interface didn't declare them. TypeScript's strict mode rejects access to undeclared properties.

```diff
  // login_page section — added at lines 76–78:
+ error_no_account: string
+ error_no_account_link: string
+ error_wrong_password: string

  // finance section — added at lines 119–126:
+ overview: string
+ overview_desc: string
+ analytics: string
+ analytics_desc: string
+ financial_data: string
+ financial_data_desc: string
+ cash_flow: string
+ cash_flow_desc: string
```

#### 2. `public/lang/en.json` — lines 24–31

**What changed:** Added the actual English translation strings for the 8 new `finance` keys.

**Why:** The `Translations` interface now requires these keys. If the JSON files don't include them, the app would crash at runtime with `undefined` values.

```diff
+ "overview": "Overview",
+ "overview_desc": "Your financial snapshot at a glance",
+ "analytics": "Analytics",
+ "analytics_desc": "Track your spending and income trends",
+ "financial_data": "Financial Data",
+ "financial_data_desc": "View and manage your financial records",
+ "cash_flow": "Cash Flow",
+ "cash_flow_desc": "Predicted income and expenses for the coming weeks"
```

Minor whitespace normalization was also applied (removed blank lines between JSON sections) — this is cosmetic and does not affect behavior.

#### 3. `public/lang/pt.json` — same structure as `en.json`

**What changed:** Added the same 8 `finance` keys with Portuguese translations.

**Why:** Same reason as `en.json` — both language files must have identical key structures.

#### 4. `components/ui/input.tsx` — line 9

**What changed:** Made the `label` prop optional.

**Why:** The `settings-dialog.tsx` component uses `<Input>` without passing a `label` prop (it uses a separate `<Label>` component instead). TypeScript errored because `label` was previously required.

```diff
  export interface InputProps extends InputHTMLAttributes<HTMLInputElement> {
-     label: string;
+     label?: string;
  }
```

#### 5. `components/cash-flow-card.tsx` — lines 28–32

**What changed:** Created a `CashFlowCardProps` interface with an optional `accountIds` prop and applied it to the component function signature.

**Why:** The dashboard page passes `accountIds` to `<CashFlowCard>`, but the component didn't accept any props. TypeScript rejected the extra prop.

```diff
+ interface CashFlowCardProps {
+     accountIds?: string[]
+ }
+
- export function CashFlowCard() {
+ export function CashFlowCard({ accountIds }: CashFlowCardProps) {
```

#### 6. `lib/logger.ts` — line 12

**What changed:** Added an intermediate `unknown` cast when extracting the `.code` property from an `Error` object.

**Why:** TypeScript does not allow directly casting `Error` to `Record<string, unknown>` because the types are structurally incompatible. Going through `unknown` first is the standard safe approach.

```diff
- const code = (err as Record<string, unknown>).code as string | undefined
+ const code = (err as unknown as Record<string, unknown>).code as string | undefined
```

#### 7. `lib/prisma.ts` — line 7

**What changed (initial):** Added `as any` to the `pool` argument passed to `PrismaPg()`.

**Why (initial):** There was a type conflict between the top-level `@types/pg` package and the version bundled inside `@prisma/adapter-pg`. The `Pool` instance created by `pg.Pool` didn't satisfy `PrismaPg`'s expected type signature. `as any` was a temporary workaround.

> Note: This was immediately improved in the next commit (see commit 2).

#### 8. `tsconfig.json` — line 33

**What changed:** Added `"scripts"` and `"prisma"` to the `exclude` array.

**Why:** The `scripts/` directory contains Node.js utility scripts (e.g. `login-test.js`) and the `prisma/` directory contains Prisma-specific files (`schema.prisma`, `seed.ts`). Both had their own type issues unrelated to the Next.js app. Excluding them from TypeScript compilation prevents their errors from blocking the build.

```diff
- "exclude": ["node_modules"]
+ "exclude": ["node_modules", "scripts", "prisma"]
```

## Commit 2 — `3ea09b2` — 2026-03-26T12:27:13

**Message:** Improve prisma.ts: pass PoolConfig directly instead of using `as any`

### Problem

Commit 1 used `as any` in `lib/prisma.ts` to work around the `@types/pg` version conflict. This silences TypeScript but hides real type errors.

### Changes

#### 1. `lib/prisma.ts` — lines 1–2, 6–7

**What changed:** Removed the `pg` import entirely and passed the connection config object directly to `PrismaPg()` instead of constructing a `pg.Pool` instance.

**Why:** `PrismaPg` accepts either a `Pool` instance or a plain config object `{ connectionString: string }`. By passing the config directly, `PrismaPg` creates the pool internally using its own bundled `pg` types — eliminating the type conflict completely.

```diff
  import { PrismaClient } from "@/lib/generated/prisma/client"
- import pg from "pg"
  import { PrismaPg } from "@prisma/adapter-pg"

  function createPrismaClient() {
-   const pool = new pg.Pool({ connectionString: process.env.DATABASE_URL })
-   const adapter = new PrismaPg(pool as any)
+   const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL })
    return new PrismaClient({ adapter })
  }
```

## Commit 4 — `157d8c4` — 2026-03-26T19:57:24

**Message:** Fix pnpm 10 build scripts and add prisma generate to build step

### Problem

Vercel uses pnpm 10.x by default. pnpm 10 blocks all dependency postinstall/preinstall scripts unless explicitly whitelisted. The Vercel logs showed:

```
Ignored build scripts: @prisma/engines@7.5.0, prisma@7.5.0, sharp@0.34.5, unrs-resolver@1.11.1
```

Without these scripts running, Prisma cannot download its query engine binary, Sharp cannot set up its native image processing module, etc.

### Changes

#### 1. `package.json` — line 7

**What changed:** Updated the `build` script to run `prisma generate` before `next build`.

```diff
- "build": "next build",
+ "build": "prisma generate && next build",
```

**Why:** Even with pnpm 10 scripts allowed, the Prisma Client code (in `lib/generated/prisma/`) must be generated before Next.js can import it. This ensures a fresh client is generated on every Vercel build.

#### 2. `package.json` — lines 89–96

**What changed:** Added the `pnpm.onlyBuiltDependencies` field listing four packages.

```diff
+ "pnpm": {
+   "onlyBuiltDependencies": [
+     "@prisma/engines",
+     "prisma",
+     "sharp",
+     "unrs-resolver"
+   ]
+ }
```

**Why:** This tells pnpm 10 to allow the postinstall/preinstall scripts for these specific packages. Without this:
- `@prisma/engines` — cannot download the Prisma query engine binary
- `prisma` — cannot run its preinstall script for version validation
- `sharp` — cannot set up native image processing (used by Next.js Image Optimization)
- `unrs-resolver` — cannot run its NAPI postinstall check

## Commit 5 — `6a6a8d5` — 2026-03-26T20:23:58

**Message:** Remove empty docs and sync route files causing "not a module" build error

### Problem

After the previous fixes resolved all TypeScript type errors, the Vercel build failed with a new error:

```
Type error: File '/vercel/path0/app/docs/page.tsx' is not a module.
```

This is because Next.js requires every `.tsx`/`.ts` file in the `app/` directory to export at least a default component (for pages) or a named export. Empty 0-byte files have no exports and are rejected.

### Changes

#### 1. `app/docs/page.tsx` — DELETED (was 0 bytes)

**Why:** This file was completely empty — 0 bytes, no content, no exports. Next.js tried to import it as a page module and failed because there was nothing to import.

#### 2. `app/docs/layout.tsx` — DELETED (was 0 bytes)

**Why:** Same issue — empty file with no exports. Would cause the same "not a module" error if Next.js processed it.

#### 3. `app/api/bank/connections/[id]/sync/route.ts` — DELETED (was 0 bytes)

**Why:** Same issue — empty API route file with no exports. Proactively removed to prevent the same build error if Next.js tried to resolve this route.

## Commit 6 — `56018f9` — 2026-03-26T20:54:46

**Message:** Fix Resend client initialization: lazy-load to avoid build-time crash when `RESEND_API_KEY` is not set

### Problem

After commits 1–5 resolved all TypeScript and empty-file errors, the Vercel build progressed further but crashed during the **"Collecting page data"** phase:

```
Error: Missing API key. Pass it to the constructor `new Resend("re_123")`
```

This happened because `lib/email.ts` created the Resend client at the **top level of the module** (line 11). During `next build`, Next.js imports modules to analyze page data — even API route modules. Since `RESEND_API_KEY` is not set in Vercel's build environment (it's a runtime-only secret), the `new Resend(undefined)` constructor threw an error.

### Changes

#### 1. `lib/email.ts` — lines 11–17

**What changed:** Replaced the top-level `Resend` instantiation with a lazy singleton pattern.

```diff
- const resend = new Resend(process.env.RESEND_API_KEY);

+ let _resend: Resend | null = null;
+ function getResend() {
+   if (!_resend) {
+     _resend = new Resend(process.env.RESEND_API_KEY);
+   }
+   return _resend;
+ }
```

**Why:** The `getResend()` function is only called when `sendPasswordResetEmail()` or `send2FACode()` actually runs at runtime — never during build. This means the `RESEND_API_KEY` environment variable is only read when emails are actually being sent (in production), not when Next.js is collecting page data during the build.

#### 2. `lib/email.ts` — line 31 (inside `sendPasswordResetEmail`)

```diff
- await resend.emails.send({
+ await getResend().emails.send({
```

#### 3. `lib/email.ts` — line 125 (inside `send2FACode`)

```diff
- await resend.emails.send({
+ await getResend().emails.send({
```

**Why:** Both email-sending functions now use `getResend()` instead of the removed top-level `resend` constant.

## Important note for Vercel deployment

After merging this PR to `main`, you need to set the `RESEND_API_KEY` environment variable in your Vercel project settings (**Settings → Environment Variables**) for the email functionality to work at runtime. The build will succeed without it, but calling `sendPasswordResetEmail()` or `send2FACode()` will fail if the key is missing.
